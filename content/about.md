<div class="hub-header">
  <p class="hub-kicker">About</p>
  <h2>구조를 다시 나누는 백엔드 개발자</h2>
  <p class="hub-lede">
    이 포트폴리오는 구현 결과를 나열하기보다, 제품 문제를 서비스 경계와 데이터 흐름의 문제로 다시 정의하는 방식을 보여주기 위해 만들었습니다.
    클라이언트 전환, 도메인 분리, retrieval 구조, AI 안전장치처럼 운영 비용이 커지는 지점을 먼저 보고 설계를 고릅니다.
  </p>
</div>

<section class="hub-section">
  <p class="hub-section-kicker">Role</p>
  <h3>한 줄 역할 정의</h3>
  <ul class="hub-list">
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">Position</span>
        <p>제품을 만들다가 생기는 구조 문제를 API, 데이터 모델, 캐시, 서비스 경계, AI 안전장치 재설계로 풀어내는 백엔드 제품개발자입니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">Focus</span>
        <p>문제를 기능 목록이 아니라 구조 전환 시점과 운영 tradeoff로 읽고, 왜 그런 선택을 했는지를 한 페이지 안에서 닫히게 설명하는 데 집중합니다.</p>
      </div>
    </li>
  </ul>
</section>

<section class="hub-section">
  <p class="hub-section-kicker">Strengths</p>
  <h3>강점 3가지</h3>
  <ul class="hub-list">
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">Strength 1</span>
        <p>제품 문제를 백엔드 구조 판단으로 바꿉니다. 클라이언트 전환, 도메인 분리, 검색 구조 재정의처럼 제품 변화가 시스템 경계를 어떻게 바꾸는지 설명할 수 있습니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">Strength 2</span>
        <p>근거를 남기는 구조를 선호합니다. 관측값, 검증 범위, 실패와 한계를 문서 안에 남겨 다음 판단의 근거로 다시 사용할 수 있게 만듭니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">Strength 3</span>
        <p>운영 비용을 같이 봅니다. 캐시, 다중 DB, AI proposal safety, traceability처럼 기능보다 유지 가능성과 위험 제어를 함께 고려합니다.</p>
      </div>
    </li>
  </ul>
</section>
