# API Surface

BagInDB의 외부 계약은 단순 관리자용 CRUD보다 탐색형 API에 더 가깝습니다. 이 문서는 "무슨 엔드포인트가 있는가"보다 "왜 그런 응답 형태를 택했는가"를 중심으로 읽어야 합니다.

## 핵심 엔드포인트 축

- 브랜드 목록과 상세
- 카테고리 구조 조회
- 제품 목록과 상세
- 필터 기반 검색

## 이 API가 해결하려던 문제

- 사용자는 장비를 쓰기보다 찾고 비교하는 시간이 더 길었습니다.
- 브랜드와 카테고리, 제품 스펙을 빠르게 넘나드는 탐색 경험이 중요했습니다.
- 그래서 생성/수정 중심 API보다 조회 응답 구조가 더 중요한 도메인이었습니다.

## 이 구조를 택한 이유

- 장비 데이터는 쓰기보다 읽기가 훨씬 많았습니다.
- 사용자는 브랜드와 카테고리, 상세 스펙을 오가며 탐색하기 때문에 조회 API의 응답 형태가 더 중요했습니다.

## 요청이 처리되는 순서

1. `routes`가 요청을 받고 파라미터와 인증을 확인합니다.
2. `handlers`가 요청별 처리 흐름을 조합합니다.
3. `cache`와 `db`가 재사용 가능한 응답과 실제 조회를 나눠 처리합니다.
4. `models`가 카테고리별 스펙을 응답 형태로 정리합니다.

## 내부 계층

- `routes`는 외부 요청을 받는 진입점입니다.
- `handlers`는 요청별 로직을 묶는 얇은 처리 계층입니다.
- `db`는 PostgreSQL 조회 로직을 구성합니다.
- `cache`는 반복 조회가 많은 응답을 앞단에서 재사용합니다.
- `models`는 카테고리별로 다른 스펙을 담는 표현 계층입니다.

## 이 문서에서 봐야 하는 핵심

- 장비 API는 관리자 기능보다 탐색 흐름을 우선해 설계됐습니다.
- JSONB 모델은 엔드포인트를 단순화하기 위한 결과이기도 합니다.
- 읽기 패턴을 먼저 본 덕분에 이후 캐시 전략과도 자연스럽게 이어질 수 있었습니다.
