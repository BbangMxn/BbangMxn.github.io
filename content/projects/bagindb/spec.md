BagInDB는 장비 API를 무작정 분리하지 않고, 먼저 어떤 조회 구조와 응답 기준이 독립 서비스의 성공 조건인지 spec으로 고정한 뒤 구현을 시작했습니다. 이 문서는 장비 도메인을 별도 백엔드로 읽어낸 이유를 가장 짧게 정리합니다.

## Problem

- 장비 데이터는 브랜드, 카테고리, 제품 스펙, 필터 검색, 다국어 표현처럼 읽기 중심 요구가 강했고, 커뮤니티 데이터와 같은 리듬으로 다루기 어려웠습니다.
- 카테고리마다 속성이 달라 정규화만으로는 응답 유연성과 탐색 속도를 동시에 맞추기 어려웠습니다.

## Goal

- 장비 탐색을 커뮤니티 서비스와 분리해도 되는지 서비스 경계 차원에서 검증해야 했습니다.
- 반복 조회가 많은 인기 구간에서 빠른 응답을 만들 수 있는 읽기 중심 API 구조가 필요했습니다.

## Decision

- 장비 도메인을 BagInCoffee와 분리한 별도 서비스로 다뤘습니다.
- 카테고리별로 달라지는 속성은 JSONB 기반 스펙 모델로 수용했습니다.
- 인기 조회는 Redis를 앞단에 두고, 낮은 자원 환경을 고려해 Rust 런타임을 선택했습니다.

## Acceptance Criteria

- 브랜드, 카테고리, 제품 목록, 상세, 필터 검색이 독립 API 표면으로 정리되어야 합니다.
- 다국어 표현과 카테고리별 스펙 차이를 한 스키마 안에서 유연하게 담을 수 있어야 합니다.
- 인기 조회와 꼬리 조회를 구분해 캐시 효과와 미스 구간을 관찰할 수 있어야 합니다.
- 장비 탐색 도메인이 커뮤니티 서비스와 다른 운영 기준을 가진다는 점이 문서와 구조 양쪽에서 설명 가능해야 합니다.

## Out Of Scope

- 관리자 CMS와 대규모 쓰기 파이프라인 완성
- 완전한 캐시 무효화 정책과 장기 운영 자동화
