BagInDB는 캐시를 부가 기능이 아니라 API 성격 자체를 바꾸는 요소로 다뤘습니다. 이 문서는 "Redis를 붙였다"는 설명보다, 어떤 조회를 빠르게 만들 가치가 있었는지 정리하는 문서입니다.

## 캐시가 중요한 이유

- 장비 목록과 상세는 반복 조회가 많았습니다.
- 인기 상품과 브랜드는 같은 요청이 계속 들어오는 구조였습니다.

## 캐시를 어디에 붙였는가

- 목록과 상세 같은 읽기 API 앞단에서 캐시를 먼저 확인합니다.
- 캐시 미스일 때만 PostgreSQL JSONB 저장소를 조회합니다.
- 다시 조회될 가능성이 높은 결과만 캐시에 적재합니다.

## 캐시 계층이 실제로 하는 일

- 먼저 Redis에서 재사용 가능한 응답을 확인합니다.
- 캐시 미스가 나면 PostgreSQL JSONB 데이터를 조회합니다.
- 다시 재사용할 만한 결과만 캐시에 적재합니다.

## 이 전략이 맞았던 지점

- 인기 브랜드와 인기 제품에서는 재사용률이 높았습니다.
- 응답 시간을 안정적으로 줄일 수 있는 축이 분명히 있었습니다.
- 장비 API가 읽기 중심 서비스라는 점을 더 분명히 확인할 수 있었습니다.

## 관찰한 한계

- 인기 상품에서는 캐시 효과가 컸습니다.
- 꼬리 조회가 많은 상품군에서는 캐시 미스가 자주 발생했습니다.
- 그래서 캐시 적중률만이 아니라 어떤 조회를 캐시 대상으로 볼지 기준을 더 세워야 했습니다.

## 다음에 더 정교해져야 하는 부분

- 무효화 기준을 더 명확히 나누는 것
- 인기 조회와 꼬리 조회를 같은 정책으로 다루지 않는 것
- 캐시 적중률 숫자보다 어떤 요청군을 빠르게 만들고 싶은지 먼저 정의하는 것
