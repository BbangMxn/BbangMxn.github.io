# Retrieval Map

Nazgul의 검색 흐름은 질의 하나를 바로 청크에 던지는 구조가 아니라, 카드와 청크, 답변, Trace를 분리하는 방식입니다. 이 문서는 연구형 파이프라인을 제품 문제와 연결해서 읽기 위한 압축본입니다.

## 입력

- 사용자의 취미 선택
- 홈 피드와 탐색 화면에서 쌓이는 관심사 맥락
- 실제 검색 질의

## 왜 입력이 중요했는가

- Nazgul은 추상 지식 검색기가 아니라 취미 기반 제품이었습니다.
- 그래서 질의만이 아니라 사용자의 취미 선택과 탐색 행동이 함께 맥락이 됐습니다.
- 이 구조 덕분에 개인화는 모델의 추측이 아니라 제품 상태에서 나온 입력이 됩니다.

## 중간 단계

- 먼저 Entity Card 후보를 찾습니다.
- 카드 안에서 더 좁은 청크 후보를 선택합니다.
- 카드와 청크를 기반으로 추천 포스트와 답변을 구성합니다.

## 카드 중심 구조를 택한 이유

- 바로 청크를 고르면 출처와 맥락이 쉽게 파편화됩니다.
- Entity Card는 취미, 카테고리, 출처를 묶은 중간 단위 역할을 합니다.
- 그래서 검색 결과를 제품 화면 위에서 더 설명 가능한 형태로 유지할 수 있습니다.

## 출력

- 추천 카드
- 연결된 포스트
- 답변 근거
- RecommendationTrace

## Trace를 남긴 이유

- 느린 검색이나 실패한 추천을 다시 읽을 수 있어야 했습니다.
- 카드 선택이 문제였는지, 청크 선택이 문제였는지 구분하려면 결과만으로는 부족했습니다.
- RecommendationTrace는 연구형 프로젝트에서 실패를 다시 분석하는 장치입니다.

## 이 구조의 의미

- 검색 결과를 설명 가능한 형태로 남길 수 있습니다.
- 실패했을 때 카드 선택이 문제였는지, 청크 선택이 문제였는지 다시 볼 수 있습니다.

## 현재까지 확인한 범위

- 로컬 환경에서 카드 3개, 추천 포스트 3개를 반환하는 흐름을 확인했습니다.
- SQLite와 Ollama 조합은 느렸지만, 병목이 어느 단계에 있는지는 더 분명하게 볼 수 있었습니다.
- 그래서 이 문서는 완성된 검색 아키텍처 설명보다, 어떤 연구 방향이 실제로 동작하기 시작했는지 보여 주는 지도 역할을 합니다.
