BagInCoffee는 커뮤니티 기능을 계속 붙이는 프로젝트로 흘러가기 쉬웠기 때문에, 먼저 무엇을 성공으로 볼지 spec으로 고정한 뒤 구현을 진행했습니다. 이 문서는 웹 프로토타입, Flutter 전환, BagInDB 분리 판단이 어떤 기준에서 나왔는지 짧게 정리합니다.

## Problem

- 웹 프로토타입만으로는 커뮤니티 핵심 흐름은 검증할 수 있었지만, 제품을 계속 키울수록 웹과 모바일을 따로 가져가는 비용이 커졌습니다.
- 브랜드, 카테고리, 제품 스펙 탐색은 커뮤니티 피드와 다른 읽기 패턴을 가져 같은 백엔드 안에 오래 두기 어려웠습니다.

## Goal

- 검증된 커뮤니티 핵심 흐름을 유지한 채 웹과 모바일 전략을 다시 선택할 수 있어야 했습니다.
- 장비 도메인을 별도 서비스로 분리해도 되는지 구조적으로 판단할 수 있어야 했습니다.

## Decision

- 초기 검증은 SvelteKit 웹 프로토타입으로 빠르게 진행하고, 이후 제품 경험 통합은 Flutter로 옮기는 2단계 전략을 택했습니다.
- 장비, 브랜드, 제품 스펙 탐색은 BagInDB로 분리하고, 커뮤니티 흐름과 장비 탐색 흐름을 다른 도메인으로 다뤘습니다.
- 인증은 Supabase를 공통 기준점으로 두고, 클라이언트와 하위 서비스를 느슨하게 연결했습니다.

## Acceptance Criteria

- 웹 프로토타입에서 검증한 커뮤니티 흐름이 Flutter 전환 뒤에도 유지되어야 합니다.
- 웹과 모바일을 하나의 제품 전략으로 설명할 수 있는 클라이언트 구조가 있어야 합니다.
- 브랜드, 카테고리, 제품 스펙 탐색이 BagInDB라는 별도 API 경계로 분리되어야 합니다.
- 클라이언트 전환과 도메인 분리의 이유가 문서와 화면 양쪽에서 설명 가능해야 합니다.

## Out Of Scope

- 완성된 상용 커뮤니티 제품 출시
- 운영 지표와 장기 유지보수 정책의 완결
