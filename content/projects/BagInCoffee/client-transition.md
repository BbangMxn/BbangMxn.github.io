BagInCoffee의 클라이언트 역사는 웹에서 모바일로 확장한 것이 아니라, 제품 검증 단계와 운영 단계에서 서로 다른 클라이언트 전략을 선택한 기록입니다. 핵심은 기술 교체가 아니라, 어떤 단계에서 어떤 도구가 맞았는지 분리해서 봤다는 점입니다.

## 웹 단계

- `BagInCoffee-Web/`은 커뮤니티 피드와 브랜드 탐색을 빠르게 검증하기 위한 SvelteKit 기반 실험이었습니다.
- 핵심은 완성도보다 속도였고, 제품의 첫 화면 흐름을 빨리 만들 수 있다는 점이 중요했습니다.

## 초기 검증

- 사용자가 피드와 콘텐츠 흐름을 어떻게 소비하는지
- 커뮤니티와 브랜드 탐색이 하나의 제품 안에서 같이 보일 수 있는지
- 제품의 핵심 경험이 무엇인지

## 전환 이유

- 웹과 모바일을 따로 관리하기보다 한 코드베이스 안에서 경험을 묶는 편이 이후 제품 확장에 더 맞았습니다.
- 커뮤니티 앱은 화면 일관성이 중요해서, 기능 추가보다 UI 리듬을 먼저 통합하는 선택이 필요했습니다.

## 유지한 것

- 커뮤니티 피드와 콘텐츠 흐름
- 브랜드 탐색이 제품 안에서 차지하는 위치
- 사용자 경험 중심의 제품 구조

## 버린 것

- 웹 프로토타입의 핵심 사용자 흐름은 유지했습니다.
- 클라이언트 구현 방식은 바꿨지만, 어떤 기능이 제품의 중심인지 판단한 결과는 그대로 이어졌습니다.

## 검증

- 웹 프로토타입은 버려진 시도가 아니라 다음 구조 판단의 근거였습니다.
- Flutter 전환은 선언이 아니라 실제로 웹과 모바일 경험을 함께 가져갈 수 있는 방향 검증이었습니다.
- 클라이언트 기술은 바뀌었지만 제품의 핵심 문제 정의는 유지됐다는 점이 중요합니다.

## 남은 질문

- Flutter 중심 전략이 실제 운영에서도 가장 단순한 선택인지
- 클라이언트 통합이 백엔드 경계와 얼마나 안정적으로 맞물리는지
- 이후 기능 확장에서 어느 시점에 다시 분리 판단이 필요할지
