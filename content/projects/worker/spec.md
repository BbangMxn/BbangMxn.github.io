Worker는 AI 기능을 많이 붙이는 프로젝트로 흐르기 쉬웠기 때문에, 먼저 AI를 어디까지 허용할지 spec으로 고정한 뒤 구현을 이어갔습니다. 이 문서는 업무 도구 안에서 제안, 분류, 동기화를 어떤 기준으로 성공으로 봤는지 가장 짧게 정리합니다.

## Problem

- 이메일, 캘린더, 연락처를 한 화면에 모으는 것만으로는 반복 분류와 정리 문제를 충분히 해결하기 어려웠습니다.
- AI가 바로 메일 전송이나 일정 생성을 실행하면 실수 비용이 커서, 자동화보다 안전한 실행 경계가 먼저 필요했습니다.

## Goal

- 업무 도구 안에서 AI가 제안, 분류, 동기화, 개인화 답장으로 이어지는 흐름을 안전하게 가질 수 있어야 했습니다.
- 모든 요청을 곧바로 LLM으로 넘기지 않고, 어떤 판단을 코드로 처리하고 어떤 구간만 AI에 맡길지 구분할 수 있어야 했습니다.

## Decision

- 위험한 작업은 바로 실행하지 않고 Proposal 단계를 먼저 두었습니다.
- 이메일 분류는 7단계 파이프라인으로 나눠 규칙 기반 처리와 LLM 처리를 분리했습니다.
- 검증 단계에서는 역할별 저장 구조를 분리해 관측 가능성을 우선했고, 이후 단순화 여지를 함께 남겼습니다.

## Acceptance Criteria

- 이메일, 캘린더, 연락처, AI 채팅이 하나의 워크스페이스 흐름으로 연결되어야 합니다.
- Gmail 동기화와 SSE 기반 상태 전송이 실제로 이어져야 합니다.
- 메일 전송이나 일정 생성처럼 위험한 작업은 Proposal 확인 단계를 거쳐야 합니다.
- 분류 파이프라인이 어떤 요청을 LLM으로 넘기고 어떤 요청을 코드로 처리하는지 설명 가능해야 합니다.

## Out Of Scope

- 완전 자율 실행형 에이전트
- Outlook까지 포함한 완전한 외부 연동 완성
