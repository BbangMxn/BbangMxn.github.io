# Classification Pipeline

Worker는 모든 메일을 바로 LLM에 보내지 않고, 규칙과 후속 AI 분류를 나눈 7단계 흐름을 실험했습니다. 이 문서는 비용 절감보다 "어떤 요청을 코드로 처리할 수 있고 어디서부터 AI가 필요한가"를 나누는 문서입니다.

## 파이프라인 목적

- 호출 비용 줄이기
- 명확한 규칙은 코드로 먼저 처리하기
- 정말 애매한 경우에만 LLM을 사용하기

## 흐름을 더 자세히 보면

1. 메일을 수신하고 외부 상태를 동기화합니다.
2. 명확한 규칙으로 1차 분류합니다.
3. 중요도와 처리 상태를 정리합니다.
4. 규칙으로 애매한 경우에만 LLM 분류를 호출합니다.
5. 이후 요약, 번역, 답장 생성을 필요에 따라 이어 붙입니다.

## 흐름

1. 메일 수신과 동기화
2. 기본 규칙 분류
3. 중요도와 처리 상태 판단
4. 필요 시 LLM 분류
5. 요약, 번역, 답장 생성

## 이 구조를 택한 이유

- 모든 메일을 LLM에 보내면 비용과 지연이 커집니다.
- 동시에 규칙만으로는 실제 업무 메일 문맥을 모두 다루기 어렵습니다.
- 그래서 규칙과 AI를 경쟁 관계가 아니라 계층 관계로 두는 편이 맞았습니다.

## 결과적으로 얻은 것

- LLM 호출량을 크게 줄일 수 있는 방향을 확인했습니다.
- 동시에 어떤 메일이 규칙 기반으로 처리 가능한지 경계를 더 분명히 볼 수 있었습니다.

## 이 문서에서 봐야 하는 핵심

- Worker의 분류 파이프라인은 단순 최적화가 아니라 제품 운영 정책입니다.
- 어떤 단계는 코드가 맡고, 어떤 단계는 AI가 맡는지 나눈 것이 핵심입니다.
- 그래서 이 문서는 AI 기능 설명이 아니라 업무 도구의 처리 기준 문서에 가깝습니다.
