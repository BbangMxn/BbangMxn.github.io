# Timeseries Pipeline

Smart Factory의 핵심은 데이터를 받는 것보다, 시간축 데이터를 어떻게 저장하고 다시 보여 줄 것인가에 있습니다. 이 문서는 시스템의 중간 이후, 즉 적재와 실시간 전송 축을 설명합니다.

## 기본 흐름

1. Collector가 장비 데이터를 받습니다.
2. Pipeline이 데이터를 정규화합니다.
3. TimescaleDB에 시계열 데이터를 적재합니다.
4. Redis와 WebSocket이 실시간 조회를 돕습니다.
5. Dashboard와 Alert가 운영자에게 상태를 전달합니다.

## 왜 시계열 중심으로 봤는가

- 이 문제는 단순 CRUD보다 시간축 조회가 더 중요했습니다.
- 최근 상태와 과거 이력을 동시에 다뤄야 했습니다.
- 저장소 선택이 곧 운영 방식 선택과 연결되는 영역이었습니다.

## 이 구조를 택한 이유

- 단순 CRUD보다 시간축 조회가 더 중요했습니다.
- 최근 상태와 과거 이력을 같이 다뤄야 해서 저장소와 전송 계층을 분리할 필요가 있었습니다.

## 실시간 경로를 따로 둔 이유

- 대시보드는 최신 상태를 빠르게 보여 줘야 합니다.
- 하지만 모든 조회를 실시간 경로로만 처리하면 과거 이력 분석과 운영 비용이 뒤엉킵니다.
- 그래서 TimescaleDB와 Redis/WebSocket 경로를 역할별로 나누는 편이 맞았습니다.

## 이 문서에서 봐야 하는 핵심

- Collector 이후의 데이터는 단순 저장이 아니라 운영자 경험으로 이어지는 흐름입니다.
- 이 파이프라인 설계가 있어야 대시보드, 알림, 이력 분석이 한 시스템 안에서 같이 설명됩니다.
- 결국 핵심은 저장소가 아니라 "시간축 데이터를 어떻게 운영 가능한 형태로 돌려주는가"입니다.
