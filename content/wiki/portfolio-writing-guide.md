<div class="hub-header">
  <p class="hub-kicker">위키 / 글쓰기</p>
  <h2>웹에서 빨리 읽히는 포트폴리오 글쓰기</h2>
  <p class="hub-lede">
    이 문서는 감각적인 문장 스타일을 정리하는 메모가 아닙니다.
    채용담당자와 실무자가 짧은 시간 안에 지원자의 역할, 강한 문제 영역, 대표 프로젝트, 읽는 경로를 파악하게 만드는 웹 글쓰기 기준을 외부 근거와 함께 정리한 문서입니다.
  </p>
</div>

<section class="hub-section">
  <p class="hub-section-kicker">원칙</p>
  <h3>웹 글쓰기에서 먼저 확인된 원칙</h3>
  <ul class="hub-list">
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">첫 화면</span>
        <p>첫 화면에서 “이 사이트가 누구를 위한 것이고, 무엇을 제공하는지”가 바로 보여야 합니다. 조직 사이트 연구에서도 방문자는 첫 몇 초 안에 정체성과 목적을 파악하길 기대했고, Plain Language와 명확한 한두 줄 요약이 중요하다고 정리됐습니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">스캔 구조</span>
        <p>웹 사용자는 줄글을 정독하기보다 스캔합니다. 그래서 제목은 앞부분에 핵심 단어를 두고, 모호한 `Introduction`, `Background`, `Summary` 같은 제목 대신 문서 역할이 드러나는 제목을 써야 합니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">정보 순서</span>
        <p>중요한 내용은 먼저 나와야 합니다. 사용자가 실제로 무엇을 알고 싶어 하는지 기준으로 위에서 아래로 배치하고, 배경 설명은 핵심 주장 뒤로 미루는 편이 효과적입니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">읽기 경로</span>
        <p>좋은 메인 페이지는 “읽는 순서”를 암시해야 합니다. 명확한 탐색 링크와 다음 문서 입구가 있어야 사용자가 길을 잃지 않고 더 깊게 들어갑니다.</p>
      </div>
    </li>
  </ul>
</section>

<section class="hub-section">
  <p class="hub-section-kicker">포트폴리오</p>
  <h3>효과적인 포트폴리오에서 반복해서 보이는 기준</h3>
  <ul class="hub-list">
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">대표 프로젝트</span>
        <p>프로젝트 수를 늘리는 것보다 잘 설명된 대표 작업이 더 중요합니다. 포트폴리오 조언과 채용 관련 조사에서는 얕은 프로젝트 여러 개보다 하나를 깊게 설명하는 편이 더 강하게 읽힌다고 반복해서 말합니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">역할과 영향</span>
        <p>각 프로젝트는 결과물보다 역할과 영향이 먼저 보여야 합니다. 문제, 맡은 범위, 핵심 판단, 결과 또는 검증 범위를 분리해서 보여주면 독자가 훨씬 빨리 이해합니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">문제 서사</span>
        <p>기술 스택으로 시작하면 독자는 구현 목록으로 읽습니다. 효과적인 포트폴리오는 문제, 해결 접근, 선택 이유, 결과를 이야기 흐름으로 먼저 보여주고, 기술은 그 뒤에서 보강합니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">빠른 요약</span>
        <p>요약은 길면 안 됩니다. 메인 페이지에서는 지원자의 정체성, 강한 문제 영역, 대표 프로젝트, 어디부터 읽을지까지 짧게 압축된 상태여야 합니다.</p>
      </div>
    </li>
  </ul>
</section>

<section class="hub-section">
  <p class="hub-section-kicker">사례</p>
  <h3>개발자 포트폴리오 사례에서 반복해서 보인 요소</h3>
  <ul class="hub-list">
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">첫 문장</span>
        <p>좋은 개발자 포트폴리오는 첫 화면에서 역할을 바로 말합니다. 분명한 헤드라인, 짧은 소개, 대표 프로젝트 입구가 한 화면 안에 같이 있습니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">선별</span>
        <p>프로젝트를 많이 보여주기보다 설명력이 큰 몇 개만 전면에 둡니다. 품질이 수량보다 중요하고, 프로젝트별 역할과 결과를 짧게 잡아 주는 구성이 반복해서 보였습니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">세부 문서</span>
        <p>강한 사례는 프로젝트 카드만 두지 않고 별도 상세 페이지를 둡니다. 문제, 역할, 해결 방식, 결과를 자세히 풀어 쓰는 전용 페이지가 있으면 설득력이 크게 올라갑니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">브랜드</span>
        <p>소개 문구와 프로젝트 설명의 톤이 일관됩니다. 화려한 연출보다 역할, 강점, 작업 방식이 같은 언어로 반복돼야 기억에 남습니다.</p>
      </div>
    </li>
  </ul>
</section>

<section class="hub-section">
  <p class="hub-section-kicker">제목</p>
  <h3>제목과 헤더에서 바로 적용해야 하는 원칙</h3>
  <ul class="hub-list">
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">핵심어 우선</span>
        <p>페이지 제목과 섹션 헤더는 중요한 단어를 앞에 둬야 합니다. 사용자는 스캔할 때 앞부분만 보고 판단하므로, 프로젝트명이나 주제어를 먼저 쓰고 역할 단어는 뒤에 붙이는 편이 더 빨리 읽힙니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">고유 제목</span>
        <p>페이지 제목은 서로 구분돼야 하고, 화면에서 보이는 제목 체계와 충돌하지 않아야 합니다. 브라우저 제목은 고유하게 유지하고, 화면 안에는 페이지 수준 제목을 하나만 두는 편이 가장 깔끔합니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">설명형 헤더</span>
        <p><code>Introduction</code>, <code>Overview</code>, <code>Summary</code> 같은 범용 제목보다, 섹션의 역할을 바로 드러내는 제목이 더 효과적입니다. 좋은 헤더는 문단을 읽지 않아도 왜 이 섹션이 있는지 설명합니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">안정된 계층</span>
        <p>H1은 페이지에 한 번만 쓰고, 본문은 H2와 H3로 내려가는 편이 읽기와 접근성 모두에 유리합니다. 제목 단계가 흔들리면 문서 구조가 약해지고, TOC와 보조기기 해석도 같이 불안정해집니다.</p>
      </div>
    </li>
  </ul>
</section>

<section class="hub-section">
  <p class="hub-section-kicker">적용</p>
  <h3>이 사이트 메인 페이지에 바로 적용할 기준</h3>
  <ul class="hub-list">
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">메인 헤더</span>
        <p>첫 화면에는 `나는 어떤 역할의 개발자인가`, `어떤 문제를 잘 푸는가`, `왜 이 포트폴리오가 백엔드 제품개발자로 읽혀야 하는가`가 두세 문장 안에 닫혀야 합니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">핵심 입구</span>
        <p>메인 페이지는 최소 세 가지 입구를 제공해야 합니다. 전체 허브, 구조부터 읽는 입구, 지원자 요약 문서처럼 목적이 다른 독자를 위한 경로가 분리되어야 합니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">프로젝트 카피</span>
        <p>프로젝트 설명은 `무엇을 만들었다`보다 `어떤 구조 문제를 어떻게 다시 정의했고 무엇을 검증했는가`를 앞에 둡니다. 현재 이 사이트가 이미 잡은 방향이므로 더 압축하고 더 선명하게 유지하면 됩니다.</p>
      </div>
    </li>
    <li class="hub-item">
      <div class="hub-note">
        <span class="hub-label">깊이 조절</span>
        <p>메인 페이지에서는 모든 배경 설명을 싣지 않습니다. 상세한 설계, 연구 메모, 기술 깊이는 하위 허브나 아키텍처 문서로 분리하고 메인 페이지는 판단과 입구에 집중해야 합니다.</p>
      </div>
    </li>
  </ul>
</section>
